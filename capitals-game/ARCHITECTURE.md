# Architecture — Capitals Guessing Game

## What this file is for

This is the first file to read before touching any part of the codebase.
It describes how every module fits together, what each one does, and — critically —
what each one does **not** do. Any single module can be read and edited in isolation
without needing to understand the rest of the project.

---

## The core idea

A daily guessing game. One mystery world capital is chosen per day (the same for all
players). The player guesses capitals — by typing or clicking on a map — and after
each guess is told the distance in km between their guess and the target. Flags and
country names are shown on every guess card. The game ends when the player guesses
correctly or exhausts their attempts.

## Layer 1 — Data

### `data/capitals.json`
The single source of truth. Contains one object per world capital.

**Shape of one entry:**
```json
{
  "capital": "Canberra",
  "country": "Australia",
  "flag": "🇦🇺",
  "code": "AU",
  "lat": -35.2809,
  "lng": 149.1300,
  "aliases": ["Australian Capital Territory", "ACT"]
}
```

**Rules:**
- No logic, no distances, no computed values.
- `aliases` allows players to search by country name, capital name, or common
  abbreviations — all matching happens in `validator.js`, not here.
- To fix or add a capital: edit this file only, then re-run `generate_distances.py`.

---

### `data/distances.json`
Pre-computed lookup table of distances between every pair of capitals.

**Shape:**
```json
{
  "Canberra→London": 16993,
  "London→Canberra": 16993,
  ...
}
```

**Rules:**
- Generated by `scripts/generate_distances.py`. Never edit by hand.
- Both directions are stored (A→B and B→A) for O(1) lookup.
- If a distance looks wrong, fix `capitals.json` and re-run the build script.

---

## Layer 2 — Build script

### `scripts/generate_distances.py`
Reads `capitals.json`, runs the haversine formula on every pair of capitals,
and writes `distances.json`. Run once and commit the output.

**Input:** `data/capitals.json`
**Output:** `data/distances.json`

**Rules:**
- Not part of the game. Never imported by any other module.
- Only needs to be re-run if capitals are added, removed, or their coordinates change.
- A model editing game logic or UI never needs to read this file.

## Layer 3 — Game logic

These modules contain zero DOM manipulation and zero fetch calls.
They can be run and tested in Node.js with no browser.

---

### `lib/daily_target.js`
Returns today's target capital. Uses today's date as a seed to pick
deterministically from `capitals.json` — every player gets the same capital on
the same day.

**In:** today's date (YYYY-MM-DD string)
**Out:** one capital object from `capitals.json`

**Rules:**
- Pure function. Same date always returns the same capital. No randomness.
- Does not know about guesses, game state, or the UI.

---

### `lib/validator.js`
Takes a raw string typed or selected by the player and returns the matching
capital object, or `null` if no match is found. Handles case-insensitive and
fuzzy matching across capital names, country names, and aliases.

**In:** string (e.g. `"australia"`, `"canberra"`, `"ACT"`)
**Out:** capital object | `null`

**Rules:**
- Pure function. No game state, no UI, no side effects.
- Test this in isolation with a list of tricky input strings.
- Matching logic lives here and nowhere else.

---

### `lib/game.js`
The core game state manager. Holds the target capital, the list of guesses so
far, the attempt count, and the win/lose status. Exposes one primary function:
`submitGuess(capitalObject)`.

**In:** capital object (from `validator.js` or `map.js`)
**Out:** updated game state object:
```js
{
  target: { ... },       // the mystery capital (revealed on game end)
  guesses: [ ... ],      // array of { capital, distance, flag, country }
  attempts: 3,           // number of guesses made
  maxAttempts: 6,
  status: "playing"      // "playing" | "won" | "lost"
}
```

**Rules:**
- No DOM. No fetch. Accepts and returns plain objects only.
- Distance is looked up from `distances.json`, not calculated at runtime.
- `index.html` calls this and passes the result to the UI modules.

## Layer 4 — Inputs

Both input modules emit the **same event** — `onGuess(capitalObject)` — and
are otherwise completely independent of each other. Either can be removed
without affecting the other.

---

### `ui/input.js`
An autocomplete text box. Filters `capitals.json` as the player types,
matching against capital names, country names, and aliases. When the player
selects a result, emits `onGuess(capitalObject)` and clears itself.

**In:** keystrokes from player
**Out:** `onGuess(capitalObject)` event

**Rules:**
- Does not know what the target capital is.
- Does not know the game state.
- Just emits a guess and resets. That is its entire job.

---

### `ui/map.js`
An SVG world map with one dot per capital, positioned using Mercator projection
of each capital's `lat` and `lng`. Clicking a dot emits `onGuess(capitalObject)`.
Accepts a `guesses[]` array to visually mark already-tried capitals.

**In:** dot click / receives `guesses[]` to colour tried dots
**Out:** `onGuess(capitalObject)` event

**Rules:**
- Same `onGuess` contract as `input.js`.
- Knows nothing about the target capital.
- Visual state (which dots are coloured) is driven by `guesses[]` passed in
  from `game.js` — the map does not manage this itself.

---

## Layer 5 — Output display

These modules are read-only. They receive data and render it. They never
modify game state.

---

### `ui/results.js`
Renders the guess history. Takes `guesses[]` and renders one card per guess
showing: flag emoji, capital name, country name, distance in km, and a heat
bar indicating relative closeness.

**In:** `guesses[]` array from `game.js`
**Out:** DOM list (re-renders on each new guess)

**Rules:**
- Never modifies `guesses[]`. Receives it, renders it, done.
- Heat bar is a percentage of max earth distance (~20,000 km).

---

### `ui/share.js`
Generates a shareable score card when the game ends. Takes `guesses[]` and
`status`, and outputs a copyable emoji string summarising the result.

**In:** `guesses[]`, `status` ("won" | "lost")
**Out:** emoji string copied to clipboard

**Rules:**
- Only activates on game end.
- No game logic lives here.

## How the pieces connect

```
daily_target.js  ──→  game.js  ←──  validator.js  ←──  input.js
                          │                         ←──  map.js
                          │
                    ┌─────┴──────┐
                    ↓            ↓
               results.js    share.js
                    ↓
               map.js (colours tried dots)
```

`index.html` is the only place that imports from multiple layers.
It wires the events together:
1. Player types or clicks → `input.js` or `map.js` emits `onGuess`
2. `index.html` passes the guess to `validator.js` → then `game.js`
3. `game.js` returns new state → `index.html` passes it to `results.js` and `map.js`
4. On game end → `share.js` is activated

## Rules for adding new features

- **New data field?** Add it to `capitals.json` only. Update this doc.
- **New input method?** Create a new file in `ui/`, emit `onGuess(capitalObject)`.
  Do not modify `game.js` or `validator.js`.
- **New display element?** Create a new file in `ui/`, accept data as a parameter.
  Do not reach into `game.js` directly.
- **New game mechanic?** Modify `game.js` only. The UI modules receive whatever
  state `game.js` returns — add to that state object and update the relevant
  display module.
- **Keep files under ~150 lines.** If a file is growing beyond that, it is
  probably doing two things and should be split.
